Branch name field — UX exploration

Six variants for how the new-workspace branch name should appear and behave. Variants 1–5 keep the surrounding modal layout intact; variant 6 reimagines the section. Each tab shows a few representative states.

1. Disclosure line — quiet by default, opens to edit

The pill row no longer hosts a text input. The auto-named branch shows up as a single inline string ("from main → ehsan/sticky-header-fix") with a quiet "change" link. Clicking it swaps the line in-place for a compact input that returns to the read line on blur. Loading and collision feedback live at the line, never inside a pill.

4. Prefix slug pill — author prefix is fixed, only the suffix is yours to type

The current pill conflates the author's prefix (ehsan/) with the editable slug. This variant splits the pill in two: a fixed ehsan/ "tag" on the left and a typed suffix on the right. Less to type, no chance of accidentally deleting the convention, and the pill width feels stable.

5. Footer summary — branch name moves to a "what will happen" strip

The branch name is a read-only consequence of the title for 90% of cases. Move it out of the input flow entirely. A small strip just above the Create button summarizes "what we're about to do": "Will create ehsan/sticky-header-fix from main · rename". Click "rename" to expand a single inline input on the strip.

6. Source sheet — full restructure (one variant takes a swing)

Drop the pill row entirely. Below the title and prompt, render a small "Source" sheet — repo, source branch, mode, new branch — as a 2-column key/value list. Each value is click-to-edit with the same chrome. Branch name no longer competes for visual weight with the title; it sits in a contained spec block alongside its peers.
