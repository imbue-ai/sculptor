Pi Provider Authentication — exploration mocks

Three end-to-end directions for Settings → Pi → Providers and the authenticated-only model picker. Authenticate is primarily an interactive pi /login (Sculptor's terminal-agent stack); disconnect uses pi /logout; credentials live in the shared ~/.pi/agent/auth.json. Each tab shows the full flow: populated list, empty state, the /login terminal, the optional paste-key path, disconnect, and the model picker (populated + empty).

Variant A — Unified connections list

Every provider lives in one flat, scannable list (like an integrations / connected-accounts page). Each row carries an identity, a status, and a single contextual action — Authenticate when disconnected, Disconnect when connected. Highest information density; the whole provider set is visible in one scroll. The pi /login terminal opens as an overlay.

Signature: one list · row-level status · overflow menu hides paste-key & per-provider extras

Variant B — “Connected” vs. “Add a provider”

Two distinct groups. Connected providers are first-class cards at the top — each shows how it authenticated, the models it unlocks, and a Disconnect action. Add a provider is a tidy grid of everything else below. Emphasizes what you already have and makes “what does this unlock?” obvious via model chips. The pi /login terminal opens in a centered modal.

Variant C — Master / detail with embedded terminal

A provider rail on the left, a detail pane on the right. Selecting a provider drives the pane, which shows status, a prominent Open pi login action, the paste-key alternative as a collapsible, and the models it provides. The pi /login terminal is embedded in the pane (not a modal), giving it the most room and keeping context. Best when there are many providers and per-provider detail matters.

Signature: rail + detail · inline terminal pane · per-provider detail & collapsibles
